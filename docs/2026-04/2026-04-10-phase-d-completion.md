## Bugs Found and Fixed

### Bug 1 ÔÇö Effects Never Ran (panels stuck after consensus)

**Symptom:** Consensus reached, `REBUILD_ALL_PANELS` effect generated by the V2 reducer ÔÇö but panels never updated on client.

**Root cause:** All 4 socket handlers (`request-join`, `start-session`, `disconnect`, `pointing`) were calling `shadowDispatch()`. Shadow dispatch is a **passive observer** ÔÇö it runs the reducer but never calls `runEffects()`. Effects were discarded.

**Fix:** Replaced all `shadowDispatch` calls with `dispatchAndRun`, which runs the reducer **and** executes its effects.

---

### Bug 2 ÔÇö Room Not Found on Every Action

**Symptom:** `[dispatch] Room 'default-room' not found` logged for every action. V2 reducer refused all actions.

**Root cause:** `shadowDispatch` internally called `roomRegistry.getOrCreateRoom()`, which lazily created the room on the first join. After switching to `dispatchAndRun` Ôćĺ `dispatch()`, the code called `roomRegistry.getRoom()` which returned `undefined` if no room existed yet.

**Fix:** Changed `dispatch()` to call `getOrCreateRoom()` instead of `getRoom()`, matching the old shadow behaviour.

---

### Bug 3 ÔÇö Wrong Panel Type (attention panel instead of speaker/listener)

**Symptom:** Consensus detected, `REBUILD_ALL_PANELS` fired, but all users received the `attention` panel instead of `speaker`/`listenerSync`.

**Root cause:** `panelConfigService.collectPanelContext()` reads live speaker from `getLiveSpeaker()` Ôćĺ `SpeakerManager`. But the V2 reducer stores the speaker as a **socketId** in `TableState.liveSpeaker`, not in `SpeakerManager`. The two stores were out of sync.

**Fix:** In `REBUILD_ALL_PANELS` effect handler (`runEffects.ts`), before calling `getPanelConfigFor`, resolve the socketId Ôćĺ displayName from `TableState.participants` and call `setLiveSpeaker(name, roomId)` to sync SpeakerManager so `panelBuilderRouter` routes correctly.

---

### Bug 4 ÔÇö Speaker Panel Missing Gliff Text Input

**Symptom:** Speaker panel rendered correctly but the "Etch to glif here..." textarea was missing.

**Root cause:** The textarea was wrapped in `{isMeLive && <textarea ...>}`. The `isMeLive` flag is set by the `live-speaker` socket event, which is emitted **before** `REBUILD_ALL_PANELS` sends `receive:panelConfig`. Due to React's async state batching, `isMeLive` was still `false` when the speaker panel first rendered ÔÇö hiding the textarea.

**Fix:** Removed the `isMeLive` guard. Since the server only sends a `speakerPanel` config to the actual live speaker, the guard was redundant and caused a race condition.

---

### Bug 5 ÔÇö Disagree Didn't Reset Panels

**Symptom:** After a listener disagreed (breaking sync), all clients remained stuck on the speaker/listener-sync panels instead of reverting to attention panels.

**Root cause:** `handleDisagree` called `setIsSyncPauseMode(false)` but never called `setLiveSpeaker(null)`. Previously `evaluateSync()` handled this automatically. After its removal, `SpeakerManager` still returned the old speaker name, so `panelBuilderRouter` kept routing to speaker/listener panels.

**Fix:** Added `setLiveSpeaker(null)` call in `handleDisagree` after clearing sync pause mode.

---

### Bug 6 ÔÇö Brain/Mouth/Blue-Select Handlers Couldn't Find the Speaker

**Symptom:** Clicking "Brain" (interrupt), "Mouth" (interrupt), or "Hear more voices" (blue select) had no effect. Logs showed `handleBlueSelectStart: no current speaker in session 123`.

**Root cause:** All 5 handlers (`handleBlueSelectStart`, `handleSelectBrain`, `handleSelectMouth`, `handleUnSelectBrain`, `handleUnSelectMouth`) identified the live speaker by scanning the V1 users map for `u.state === "speaking"`. In V2, the speaker's state remains `"regular"` ÔÇö only `SpeakerManager` knows who's live. Previously `evaluateSync()` set `user.state = "speaking"` on the consensus winner. With `evaluateSync` removed, no code ever set this state.

**Fix:** All 5 handlers now import `getLiveSpeaker()` and look up the speaker by name instead of by state.

## Architecture After Today

```
Client (Socket.IO)
    Ôćô socket events (pointing, clientEmits, etc.)
socketHandler.ts
    Ôćô routeAction() Ôćĺ V1 handlers (gesture state, panel emits)
    Ôćô dispatchAndRun() Ôćĺ V2 reducer Ôćĺ effects Ôćĺ runEffects()
                                               Ôćô REBUILD_ALL_PANELS
                                               Ôćô setLiveSpeaker() Ôćĺ SpeakerManager
                                               Ôćô getPanelConfigFor() Ôćĺ panelBuilderRouter
                                               Ôćô receive:panelConfig Ôćĺ each client
```

**Key principle:** V2 owns consensus detection, session phase, and speaker identity. V1 handlers still manage gesture state transitions and per-user panel personalisation. The two systems are synced at `REBUILD_ALL_PANELS` time via `setLiveSpeaker()`.

## Known Issues / Still to Test

The following scenarios have bugs or have not yet been tested:

- **Mid-session join** ÔÇö A user joining while a session is already in LIVE_SPEAKER phase. V2 sends them a `JOIN_SESSION` action which correctly lands them in `ATTENTION_SELECTION` phase in the reducer, but what panel config they receive and whether the speaker's panel updates correctly is untested.

- **Mid-session leave (non-speaker)** ÔÇö A listener disconnecting during LIVE_SPEAKER phase. V2 handles `DISCONNECT` but the effect on V1's users map and pointer cleanup has not been verified.

- **Speaker disconnect** ÔÇö When the live speaker disconnects, V2 keeps them as a ghost and holds the mic. The timeout/cleanup logic and panel reset for remaining users needs testing.

- **Pass the Mic full flow** ÔÇö `PASS_MIC`, `ACCEPT_MIC`, `DECLINE_MIC` are implemented in the V2 reducer but the connected V1 handlers (`handlePassTheMic`, `handleAcceptMicOfferFromPassTheMic`, etc.) still reference `user.state === "speaking"` in some places ÔÇö same class of bug as fixed today.

- **Drop the Mic flow** ÔÇö `handleDropTheMic` may have related `state === "speaking"` lookups; needs audit.

- **Multi-room (2 tables)** ÔÇö The original goal for after Phase D. Room isolation in V2 is implemented but has not been load-tested with two concurrent rooms.

- **Session timer expiry** ÔÇö `TIMER_EXPIRED` is implemented in the reducer but the panel reset and client notification flow hasn't been tested live.

- **Reconnect flow** ÔÇö `RECONNECT` is implemented in the reducer (`EMIT_FULL_STATE_TO_USER` effect) but the client-side handling of `v2:full-state` hasn't been validated.
